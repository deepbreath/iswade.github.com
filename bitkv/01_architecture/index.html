    BitKV 架构设计

    BitKV 被设计成一个分布式的 K-V 数据库集群，并且提供事务支持。

    BitKV的设计目标

    BitKV 被设计的主要目标包括：

    1. 生活更加美好：于操作人员的高度自动化，对开发人员来说简单易懂。
    2. 提供强一致性：提供分布式事务，没有最终一致性，或者过时读取。
    3. 提供高可用性：所有节点都可以读写，不会产生冲突。
    4. 易部署易扩展：简单的命令实现部署和弹性扩容。

    词汇表

    术语

    术语 定义
    Cluster 集群：作为一个独立运行的单一逻辑实体，对外提供服务。
    Node 节点：运行 BitKV 的单个机器，许多节点组合到一起形成集群。
    Range 范围：BitKV 存储所有用户数据和系统数据都保存在巨大的排序的 k-v 映射表中。键空间被分割为多个范围：键空间的连续块，所以每个键仅存在于单个范围中。当越来越多的 kv 保存到范围中之后，范围增长到 64MB 之后，会分割为两个范围。
    Replica 副本：BitKV 复制每个范围（默认 3 次）保存到不同的节点上。
    Leaseholder 租约保持者：对于每个范围，有一个副本持有范围租约。这个部分就称之为租约保持者。这个副本负责协调对这个范围的所有读取和写入。
    Raft Leader 对于每个范围，一个副本是写请求的 leader。通通过 Raft 协议，这个副本保证多数副本（leader 和足够多的 follower）基于 Raft 对写入提交达成一致。 Raft leader 几乎总是与租约保持者是同一个副本。
    Raft Log 对于每个范围，对范围的按时间排序的写入日志，副本已经达成一致。这个日志存在于磁盘上，并且是范围的一致性复制的真正来源。

    概念

    BitKV 依赖下面的概念，熟悉这些概念能帮助我们了解架构设计所实现的目标。

    术语 定义
    Consistency BitKV 在 ACID 语义和 CAP 定理的意义上使用“一致性”，尽管不是正式定义的那样，但是我们试图用这个术语来表达数据是无异常的。
    Consensus 当一个范围接受到写入请求后，包含范围的多数副本确认写入。这表示数据被安全地存储，并且多数节点对数据库的当前状态达成一致，即使节点中的一些离线。写入没有达成共识，转发进程会停止以保持集群内的一致性。（CP）
    Replication 复制涉及创建和分发数据副本，以及确保副本保持一致。
    Transactions 在数据库上执行的一组操作，满足 ACID 语义的要求。
    Multi-Active Availability 我们基于共识的高可用性概念，允许集群中的每个节点处理对存储数据子集的读写（基于每个范围）。 这与主动 - 被动复制形成对比，在主动 - 被动复制中，主动节点接收100％的请求流量，以及主动 - 主动复制，其中所有节点都接受请求，但通常无法保证读取都是最新的并且快速。

    概览

    BitKV 通过命令 bitkv start 启动节点，通过命令 bitkv join 加入集群。

    bitkv 进程运行之后，就可以通过客户端 bitkv 与之交互。由于节点是对称的，可以将命令发送给任何一个节点处理，这保证 bitkv 很容易与负载均衡器集成。

    接受到客户端的 RPC 消息之后，节点将其转换为对我们分布式 k-v 存储的操作。集群处理数据时，BitKV会在算法上开始在节点之间分配数据，将数据分解为我们称之为范围的64MiB块。每个范围都复制到至少3个节点，以确保高可用性。只要将数据复制到其他节点，如果节点发生故障，您仍然可以获得可用于读取和写入的数据副本。

    如果节点收到读取或写入请求，无法直接服务，它只是找到可以处理请求的节点，并与之通信。这样您就不需要知道数据的存在位置，CockroachDB会为您跟踪数据，并为每个节点启用对称行为。

    对范围内的数据所做的任何更改都依赖于共识算法，以确保其大多数副本同意提交更改，确保行业领先的隔离保证并为您的应用程序提供一致的读取，无论您与哪个节点进行通信。

    最终，使用高效的存储引擎将数据写入磁盘并从磁盘读取数据，该引擎能够跟踪数据的时间戳。这样可以支持按时间读取历史数据的功能。

    然而，虽然这个高层概述为您提供了BitKV的功能概念，但了解 tikv 进程如何对这些需求进行处理将使您更好地理解我们的架构。

    在，BitKV 将客户端的语句转换为 kv 数据，在节点之间分布，写入到磁盘。我们的架构是我们实现这一目标的过程，表现为许多层与它们上面和下面的层相互作用，并且作为相对不透明的服务。

    以下描述了每个层执行的功能，但大多数情况下忽略了其他层的细节。这种描述适用于本层，层通常将其他层视为黑盒API。 层之间存在相互作用，这些相互作用没有明确表达，需要了解每一层的功能才能理解整个过程。

    层 顺序 目的
    Coordinator 1 1. 事务处理：允许对多个范围做分布式事务。
    2. 数据分布：将复制 KV 范围表示为单个实体。
    Replication 2 1. 写入处理：在多个节点上一致且同步复制 KV 范围。
    2. 该层还能通过租约实现一致的读取。
    Storage 3 在磁盘上写入和读取 KV 数据。

    协调者

    最小化协调或阻止同时执行的操作之间的通信是最大化数据库系统中的可伸缩性，可用性和高性能的关键。

    事务处理

    单 range 事务，多 range 事务

    优化的两阶段提交协议。

    写和读（阶段 1）

    写：事务处理写入操作时，并不是直接写入到磁盘，创建了两个东西来实现分布式事务：

    1. transaction record：保存在第一个写入发生的 range 中，包含了事务的当前状态（从PENDING开始，以COMMITED或者ABORTED结束）
    2. 一个事务的所有写入的 write intents 表示代表临时的，未提交的状态。它们与标准多版本并发控制（MVCC）值基本相同，但也包含指向存储在集群上的 transaction record 的指针。

    当 write intents 创建后，BitKV 检查新的已经提交的值，如果存在，事务重启，对于相同 keys 的已经存在的 write intents，通过事务冲突解决。

    读：如果事务没有中止，则开始执行读取操作。如果读取到标准的 MVCC 值，一切都很完美。但是如果遇到任何 write intents，则操作必须通过事务冲突来解决。注释：如果直接读取老版本，保证快照隔离是不是就可以提升性能了呢？

    提交（阶段 2）

    BitKV 检查运行事务的 record 检查是否其被中止。如果被中止，则重启事务。

    如果事务通过这些检查，则变更为提交状态，将事务成功的应答发送给客户端。

    清理（异步阶段 3）

    在事务处理完后，所有的write intents 也需要被处理。协调者需要保持对所有写入 key 的跟踪，直到值满足：

    • 通过移除指向 transaction record 的信息，将 write intents 处理为 MVCC 值。
    • 删除 write intents。

    这仅仅是一个优化，如果后续的操作遇到 write intents，总是检查它们的 transaction records，通过任何操作都可以处理或者移除 write intents。

    Time

    在处理事务的过程中，协调者节点使用 HLC 时间获取一个时间戳。时间戳被用来跟踪值的版本(MVCC)，并且提供了事务隔离保证。当节点发送请求给其它节点时，会包含本地 HLC 时间戳。节点收到请求时，通知本地 HLC收到了发送来的时间戳，这对于保证节点上读/写的所有数据的时间戳小于下一个HLC时间非常有用。这还保证了负责 range 读取的节点保证事务读取的数据 HLC 时间大于 MVCC 值的时间戳（读取总是发生在写入之后。）

    如果一个读取事务的时间戳小于遇到的值的时间戳，读取事务需要重试。

    时间戳缓存

    为了保证可串行化，每当操作读取值时，我们将操作的时间戳存储在时间戳缓存中，该缓存显示正在读取的值的高水位线。每当发生写入时，都会根据时间戳缓存检查其时间戳。 如果时间戳小于时间戳缓存的最新值，我们会尝试将其事务的时间戳推送到以后的时间。 推送时间戳可能会导致事务在事务的第二阶段重新启动（请参阅读取刷新）。
