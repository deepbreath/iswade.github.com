    优化器实现 I

    查询优化

    对于一个给定的查询，找到一个正确的代价最低执行计划。

    查询优化是 DBMS 中最难实现好的部分（被证明是 NP-完全的）。

    没有优化器能真正生成“最优”的计划。

    • 使用估计技术猜测实际的计划代价。
    • 使用启发式来限制搜索空间。

    架构概览

    SQL重写

    解析器（SQL query \rightarrow​ AST）

    绑定（使用系统表，将AST \rightarrow 带有注释信息的 AST）

    AST重写

    优化（代价估计，带有注释信息的 AST \rightarrow 物理计划）

    逻辑物理计划

    优化器生成了从逻辑代数表达式到优化的物理代数表达式之间的 等价映射。

    物理算子定义了指定的特定访问路径的执行策略。

    • 依赖处理数据的物理格式。（例如，排序和压缩）
    • 不总是1对1的从逻辑和物理计划映射。

    关系代数等价

    两个关系代数表达式等价：如果在每个合法数据库实例上，这两个表达式生成相同的元组集。例如：A \bowtie (B \bowtie C) = B \bowtie (A \bowtie C)​

    观察结论

    对于 OLTP 查询优化相对容易，因为是可优化搜索：

    • 通常仅仅找出最好的索引。
    • 连接几乎总是与小基数的外键关联。
    • 可以使用简单的启发式实现。

    这里我们主要关注 OLAP 查询。

    代价估计

    对于数据库的当前状态，生成一个执行计划的代价的估计。

    • 与 DBMS 中其它任务的交互。
    • 中间结果的大小。
    • 算法选择，访问方法。
    • 资源利用（CPU、I/O、网络）
    • 数据属性（倾斜、排序、放置）

    设计决策

    1. 优化粒度
    2. 优化时间
    3. 准备语句
    4. 计划的稳定性

    优化粒度

    1. 选择1：单查询
    2. 更小的搜索空间。
    3. DBMS 不能跨查询重用结果。
    4. 为了解决资源争用问题，代价模型必须考虑当前正在运行的内容。
    5. 选择2：多查询
    6. 如果有许多相似的查询会更有效率。
    7. 搜索空间比较大。
    8. 对于扫描共享更有用。

    优化时间

    1. 选择1：静态查询
    2. 执行前选择最佳计划。
    3. 计划质量取决于代价模型的准确性。
    4. 可以使用预备好的语句分摊执行。
    5. 选择2：动态优化
    6. 在查询执行时即时选择算子计划。
    7. 将重新优化多次执行。
    8. 很难实现/调试（非确定）。
    9. 选择3：混合优化
    10. 使用静态算法编译。
    11. 如果估计的错误 > 阈值，重新优化

    准备语句

    1. 选择1：重新优化
    2. 每次查询被调用时，重新运行优化
    3. 重复使用现有计划作为起点。
    4. 选择2：多个计划
    5. 对多个参数值生成多个计划。
    6. 选择3：平均计划
    7. 选择一个参数的平均值，对所有的调用都使用。

    计划的稳定性

    1. 选择1：Hints
    2. DBA 给优化器提供 hints。
    3. 选择2：固定的优化器版本
    4. 设置优化器的版本号，一个个迁移到新的查询优化器。
    5. 选择3：后向兼容的计划
    6. 保存老版本查询计划，然后提供给新的DBMS。

    优化查找策略

    • 启发式
    • 启发式 + 基于代价的连接顺序
    • 随机算法
    • 分层搜索
    • 统一搜索

    1. 基于启发式的优化

    定义静态规则，可以将逻辑的算子转换为物理计划。

    • 尽早执行最严格的选择
    • 在连接前执行所有选择
    • 谓词/Limit/投影下推
    • 基于基数的连接顺序

    例如：INGRES 和 Oracle 的原始版本（直到1990s 中期）

    步骤1： 分析成单个变量的查询

    步骤2： 从 Q1 到 Q3 到 Q4 替换值

    优势：

    • 很容易实现和调试
    • 对于简单查询工作得很好，并且速度很快

    劣势：

    • 依靠预测计划决策效果的魔术常数
    • 当算子有很复杂的内部依赖时，几乎不可能生成好的计划

    2. 启发式 + 基于代价的连接查找

    使用静态的规则执行最初的优化。

    使用动态规划决定表的连接顺序。

    • 第一个基于代价的优化器
    • 使用分治查找方法的自底向上计划（正向链接）

    例如：System R，早期的 IBM DB2，大部分开源 DBMS

    SYSTEM R 优化器

    将查询分解成块，对每个块生成逻辑算子。

    对每个逻辑算子，生成一系列物理算子来实现。

    • 所有连接算法和访问路径的组合。
    • 然后迭代地构造一个“左深”树，最小化执行计划的估计工作量。

    步骤1： 对每个表选择最合适的访问路径。

    步骤2： 对表枚举所有可能的连接顺序。

    步骤3： 确定最小代价的连接顺序。

    优势：

    • 通常可以找到合理的计划而无需进行详尽的搜索。

    劣势：

    • 与仅使用启发式方法有相同的问题。
    • 左深 连接树并不总是最优的。
    • 必须考虑代价模型中数据的物理属性（例如排序顺序）。

    3. 随机算法

    对查询的所有可能（有效）计划的空间执行随机游走。

    持续搜索直到达到一个代价阈值或者优化器运行了固定一段时间后。

    例如：Postgres 的遗传算法。

    模拟退火

    使用一个由基于启发式生成的查询计划开始。

    计算算子的随机排列（例如交换连接顺序）

    • 总是接受一个减少代价的变更。
    • 概率接受一个代价增加的变更。
    • 拒绝所有违反正确性的变更。

    Postgres 的优化器

    更复杂的查询使用选择连接顺序的遗传算法。

    在每轮开始时，生成查询计划的不同变体。

    选择成本最低的计划并将其与其他计划进行置换。 重复。

    • 变异函数仅需要生成有效的计划。

    随机算法的优势：

    • 在搜索空间随机跳跃，可能会使得优化器跳出局部最优。
    • 更少的内存消耗（如果不需要保存历史）。

    劣势：

    • 很难确定为什么 DBMS 会选择特定的计划。
    • 必须做额外的工作保证查询计划是确定的。
    • 仍然需要实现正确的规则。

    观察结论

    使用过程语言编写查询转换规则很困难，并且容易出错。

    • 如果没有运行大量的模糊测试，没有很容易的办法来验证规则的正确性。
    • 每个逻辑算子的物理算子的生成与关于查询的更深层语义分离。

    一个更好的方法是使用声明式的 DSL（领域专用语言）编写转换规则，然后让优化器在优化阶段强制执行它们。

    优化器生成器

    使用一个规则引擎，运行转换修改查询计划算子。

    数据的物理属性嵌入到算子中。

    1. 选择1：分层搜索。优化有多个阶段完成。
    2. 选择2：统一搜索。仅执行一次查询优化。

    5. 分层搜索

    首先使用转换规则重写逻辑查询计划。

    • 引擎在应用规则前检查转换是否被允许。
    • 代价在这一步不会绝不会被考虑。

    然后执行基于代价的搜索，将逻辑计划映射到物理计划。

    STARBURST 优化器

    使用**声明式规则**的 System R 优化器的更好的实现。

    阶段1：查询重写。计算查询的SQL块级关系演算查询表示。

    阶段2：计划优化。执行 System R 风格的动态规划阶段。

    例如，最新版本的 IBM DB2。

    优势：

    • 实际工作很好，并且性能好。

    劣势：

    • 很难为转换分配优先级。
    • 如果不计算多个成本估算，很难评估某些转换。
    • 规则维护很痛苦。

    6. 统一搜索

    统一逻辑到逻辑和逻辑到物理的转换。

    • 没有必要区分阶段，因为所有的都是转换。

    这个方法生成了很多的转换，所以它大量使用了备忘信息来减少冗余的工作。

    VOLCANO 优化器

    通用的基于代价的查询优化器，基于代数规则的等价性。

    • 很容易增加新的操作和等价规则。
    • 在规划期间将数据的物理属性作为第一类实体。
    • 使用分支定界搜索的自顶向下的方法（后向链表）。

    例如，学术原型。

    自顶向下 vs. 自底向上

    自顶向下优化

    • 从最终的期望的结果开始，然后向下找到最优的计划。
    • 例如：Volcano，Cascades

    自底向上优化

    • 开始的时候什么都没有，然后向上构建计划，得到最终期望的结果。
    • 例如：System R，Starburst

    VOLCANO 优化器

    从我们想要的一个查询的逻辑计划开始。

    调用规则生成新的节点和遍历树。

    可以创建需要输入以具有某些属性的“强制执行器”规则。

    优化器需要枚举所有可能不重复的转换。

    从逻辑到物理尽可能快，尝试所有可能的计划。

    • 使用自顶向下规则引擎，执行分支定界剪枝。
    • 使用备忘录缓存等价的算子。

    优势：

    • 使用声明式的规则生成转换。
    • 高效的搜索引擎更好的扩展性。使用备忘录兼容冗余的评价。

    劣势：

    • 在优化搜索之前，所有等价的类被往前展开成所有的逻辑算子。
    • 不是很容易修改谓词。
