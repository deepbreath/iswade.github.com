    查询优化策略

    1. 启发式：INGRES，Oracle（直到90年代中期）
    2. 启发式+基于代价的连接：System R，早期 DB2，大部分开源 DBMS
    3. 随机搜索：80年代学术，当前的 Postgres
    4. 分层搜索：IBM 的 STARBURST（80年代后期），当前的 DB2，Oracle
    5. 统一搜索：90年代的Volcano/Cascades，当前的 MSSQL + Greenplum

    Postgres 优化器

    为查询优化强加了一个严格的流程：

    • 第一个阶段执行启发式规则的重写。
    • 然后执行基于代价的搜索找到优化的连接顺序。
    • 所有其它的部分都被当做“附件”。
    • 然后递归下降执行子查询优化。

    很难修改或者扩展，因为顺序必须保留。

    优化器生成器

    允许 DBMS 实现者对优化查询编写声明式规则框架。

    • 将搜索策略和数据模型分离。
    • 将转换规则和逻辑算子到物理规则和物理算子分离。

    实现可以独立于优化器的搜索策略。

    例如：Starburst, Exodus, Volcano, Cascades, OPT++

    分层搜索

    首先使用转换规则重写逻辑查询计划。

    • 引擎检查在引用前检查是否被允许。
    • 在这一步不考虑代价。

    然后执行基于代价的搜索将逻辑计划映射到物理计划。

    统一搜索

    统一从逻辑到逻辑和逻辑到物理转换的概念。

    • 不需要分离各个阶段，因为所有的都是转换。

    这个方法生成了更多的转换，所以重度依赖备忘来减少冗余的工作。

    CASCADES 优化器

    Volcano 查询优化器的面向对象的实现。

    可以通过直接映射函数对简单表达式重写而不是通过穷举搜索。

    CASCADES 实现

    独立的：

    1. Wisconsin OPT++(1990s)
    2. Portland State Columbia(1990s)
    3. Pivotal Orca(2010s)
    4. Apache Calcite(2010s)

    集成的：

    1. Microsoft SQL Server(1990s)
    2. Tandem NonStop SQL(1990s)
    3. Clustrix(2000s)
    4. CMU Peloton(2010s)
