    基于代价的优化

    生成对于当前数据库状态的执行一个特定查询计划的代价评估。

    • 评估仅对数据库内部有意义。

    对于查找策略，是独立的。

    代价模型组成

    1. 物理代价
    2. 预估CPU周期，I/O，缓存不命中，RAM消耗，预取等等。
    3. 依赖硬件。
    4. 逻辑代价
    5. 估计每个算子的结果集大小。
    6. 独立的算子算法。
    7. 对算子结果集大小需要做估计。
    8. 算法代价
    9. 算子算法实现的复杂度。

    基于磁盘的 DBMS 代价模型

    磁盘访问的次数总是占查询执行时间的主要部分。

    • CPU 代价几乎可以忽略。
    • 需要考虑顺序和随机 I/O。

    如果 DBMS 对缓存管理完全控制，就很容易建模。

    • 我们知道替换策略，固定方式，对磁盘的独占访问。

    Postgres 代价模型

    使用 CPU 和 I/O 代价的组合，通过常量因子来加权。

    默认设置对于没有大量内存的磁盘驻留数据库比较友好。

    • 处理在内存中个一个元组比从磁盘上读取快 400 倍。
    • 顺序读 I/O 比随机读快4倍。

    IBM DB2代价模型

    在系统表中的数据库特征。

    硬件环境。

    存储设备特点。

    通信带宽（仅分布式）

    内存资源（缓存池，排序堆）

    并发环境：平均用户数量，隔离级别/阻塞，可用锁数量

    内存 DBMS 代价模型

    没有 IO 代价，但是我们需要计算 CPU 和内存访问代价。

    内存代价很难计算因为 DBMS 没有控制缓存管理。

    • 未知的替换策略，没有固定、共享缓存，非统一的内存访问。

    每个算子处理的元组数量对 CPU 代价而言是一个合理的估计。

    小型代价模型

    两阶段模型自动生成从逻辑模型生成硬件模型。

    阶段1：指定执行原语。

    • 当执行一个查询时 DBMS 的操作列表。
    • 例如，估计谓词，索引探测，排序。

    阶段2：微测评

    • 在启动时，配置操作来计算 CPU 内存代价。
    • 这些测量用在公式中计算基于表大小的算子代价。

    观察结论

    每个算子处理的元组数量依赖下面三个因素：

    1. 每个表可用的访问方法。
    2. 在数据库中属性中值得分布。
    3. 查询中使用的谓词。

    简单查询很容易计算，复杂的查询不容易评估。

    选择度

    一个算子的选择度是每个谓词访问数据的百分比。

    • 建模为是否满足任何给定元组的谓词的概率。

    DBMS 计算选择度使用：

    • 域约束
    • 预先计算的统计数据（区域地图）
    • 直方图/近似值
    • 采样

    IBM DB2 - 学习优化器

    DBMS 在正常处理查询过程中扫描一个表时更新表的统计信息。

    检查优化器的评估是否与实际数据中遇到的值相匹配，并逐步更新它们。

    近似

    维护准确的统计信息代价很高速度很慢。

    使用近似的数据结构 草图 生成限制误差的估计。

    • count distinct
    • 分位数
    • 频繁访问条目
    • 元组草图

    Yahoo! Sketching Library

    采样

    对目标数据集的随机样本上执行一个谓词。

    元组的个数依赖表的大小。

    方法1：维护只读副本

    • 周期性刷新来维护准确性。

    方法2：采样实际表

    • 使用读未提交隔离级别。
    • 可能会读取相同逻辑元组的多个版本。

    结果基数

    每个算子将生成的元组数量是根据其**选择性**乘以其输入中的**元组数**来计算的。

    假设＃1：统一数据 →值的分布是相同的。

    假设＃2：独立谓词 →属性的谓词是独立的

    假设＃3：包含原则 →连接键的域重叠，使得内部关系中的每个键也将存在于外部表中。

    学习到的内容

    查询优化比快速引擎更加重要。

    • 基于代价的连接顺序是必须的。

    基数估计通常是错误的。

    • 尝试使用不依赖估计的算子。

    Hash 连接 + 顺序扫描是一个可靠的执行模型。

    • 可用的索引越多，计划就越脆弱（但平均速度也更快）。

    使用精确模型是浪费时间。

    • 更好地改进基数估算

    部分思考

    使用处理的元组数是内存DBMS的合理成本模型。

    • 但如何计算这个并不简单。

    我认为采样+草图的组合是实现准确估算的方法。
