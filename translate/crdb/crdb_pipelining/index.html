    管道化共识写入加速分布式 SQL 事务

    分布式事务

    1. 存储
    2. 复制
    3. 分布
    4. 事务：preparation, commit, cleanup

    分布式事务的开销

    主要的延迟开销是：存储延迟和复制延迟。

    L_{txn} = L_{prep} + L_{commit}

    L_{prep} = R * L_r + W * L_w \simeq W * L_w = W * L_c

    L_{commit} = L_c

    L_{txn} = (W + 1) * L_c

    L_c​: 单位延迟

    现有技术

    v1.0 并行语句执行；通过立即返回使得客户端可以执行下一条语句

    写入缓存直到提交

    对竞争的事务产生负面影响。

    write intent 类似于传统数据库的行锁，通过在事务的生命周期中“获取”这些锁并允许其他事务的读取在过渡期间创建读写冲突，我们在运行TPC-C等工作负载时观察到事务中止的大幅上升。事实证明，在事务结束时执行所有写入（即获取所有锁定）会使较弱的隔离级别（如快照隔离），因为此类隔离级别允许事务的读取时间戳和其写入时间戳分开。

    异步共识

    在写入过程中，只需要提交 Raft 写即可，不需要等待成功就可以立即返回。这样就可以忽略 W * L_c

    验证 intent 写入

    提交事务之前，需要等待所有的共识写入完成。所有 intent 成功之后，事务可以将 record 中的开关变为 COMMIT。

    事务管道化会影响我们的延迟模型。

    L_{txn} = L_{commit} = 2 * L_c

    下一步：并行提交 允许在事务提交时发生的两个操作，验证 intent 写入和变更记录上的开关并行运行。

    注：CockroachDB 已经把乐观并发控制变为悲观控制。
