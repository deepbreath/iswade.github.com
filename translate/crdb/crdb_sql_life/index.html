    [译] SQL的一生（CRDB）

    介绍

    这篇文章旨在解释 CockroachDB 中 SQL 查询的执行流程，解释系统中经过各个层的代码路径（网络协议、SQL session 管理、解析、执行计划、语法树转换、查询运行、KV 代码的接口、KV 请求的路由、请求处理、Raft、磁盘存储引擎）。主要想法是给出各类组件结构的高层统一视图；不会深入探讨某一个特定的部分，但是会给出其它的一些文档。以及相关的代码。

    这篇文档不会讨论设计决策，会聚焦在跟踪当前实际代码上。

    这篇文档主要面向的读者是那些想深入了解一个现代的、年轻的不同于设计文档所描述数据库，也希望对开源贡献者和 CockRoachDB 的新的工程师有所有帮助。

    SQL 执行器

    sql.Executor 负责解析语句、执行然返回结果给 pgwire.v3Conn。入口点是 Executer.execRequest，这个接口接收的是 string 类型的一批语句。语句的执行过程在 sql.Session 对象的上下文中完成，这个对象聚集了连接的状态信息（例如：选择的数据库、可以设置的各种变量、事务状态），也会在任意给定的时间统计这个连接内存使用情况。Executor 也会操纵一个 planner 结构，这个结构提供了用来规划和执行一个查询请求的主要功能。

    Executor.execRequest() 实现了从 pgwire 批量接受语句的状态机，一条条地执行，更新 Session' 的事务状态（是否一个新的事务开始了，或者一个旧的事务结束了，是否当前遇到了一个错误必须要退出当前事务）并且会返回结果和控制信息给 pgwire。从客户端会继续接收到下一批语句会从上一批的事务状态继续。

    解析

    执行器第一件做的事情是解析语句，解析过程用到了 yacc 类似的语法文件，由go-yacc 生成的 LALR 解析器，最初是从 Postgres 拷贝并剥离下来，然后随着越来越多的 SQL 支持慢慢增长。解析过程将 string 转换成一个由抽象语法树（AST）组成的数组，一个语句一条记录。AST 节点是结构体定义在 sql.parser 包中，一般是两种类型 ：语句和表达式。表达式实现了通用接口方便应用树转换。这些 AST 会被 planner 转换为执行计划。

    执行

    当前有一个语句的列表，Executor.execRequest() 按顺序遍历，一次执行一个事务中的语句（例如：语句在 BEGIN 和 COMMIT/ROLLBACK 之间的一组语句，或者在事务之外的一条语句）。如果一个会话在前一批执行执行完之后事务被开启，会继续后面的语句直到遇到一个 COMMIT/ROLLBACK。“继续执行语句”由 runTxnAttempt 完成；这个函数会返回到执行时遇到 COMMIT/ROLLBACK。
